# Schema-First Config: Implementation Plan

**Supersedes:** `NATIVE_PYDANTIC_CONFIG_PLAN.md` (which proposed deleting schema files — we keep them).  
**Refines:** `CONFIG_RESTRUCTURE_PLAN.md` (layout migration done; this plan covers tooling + type safety).  
**Status:** Plan — not yet implemented.

---

## 1. Design Principles

1. **`config_schema.json` is the single source of truth.** It is a transmittable, inherently serializable artifact for the distributed system. It is **hand-maintained**, never generated, never deleted.
2. **Pydantic models are generated from the schema** via `datamodel-code-generator`. They provide development-time type safety, IDE autocomplete, and runtime validation. They are a **build artifact** — never hand-edited.
3. **Services do NOT fetch the entire `AppConfig` object.** They use nested `ConfigKeys` path values (derived from the schema's dot-paths) to request **only the section they need** via `config_api.aget(ConfigKeys.services.tts)`. The return type is validated against the corresponding Pydantic sub-model.
4. **One command keeps everything in sync:** `make generate-config` regenerates models, keys, and defaults from the schema. CI enforces that generated files match the committed schema.

| Artifact | State | Problem |
|----------|-------|---------|
| `config_schema.json` (1255 lines) | **Old layout** (`general`, `plugins`, `mcp`, `gateway`, `ui`) | Does NOT match `config_defaults.json` or runtime code. Must be rewritten to `services.*` layout. |
| `config_defaults.json` (294 lines) | **New layout** (`ui`, `system`, `services.*`) | Correct structure but hand-maintained; should be generated from schema defaults. |
| `app/shared/config/models.py` (350 lines) | Auto-generated by `datamodel-codegen` | Header says "from config_schema.json" but shape matches defaults, not on-disk schema. Stale. |
| `app/shared/config/models/app_config.py` (354 lines) | **Hand-written** Pydantic v2, `extra='ignore'`, with defaults | Used by `ConfigManager._validate_config()`. Should be replaced by codegen output. |
| `app/shared/config/keys.py` (67 lines) | Hand-written `ConfigKeys(str, Enum)` | Correct paths for `services.*`; 1 stale key. Should be generated. |
| `ConfigAPI.aget()` | Returns raw `Any` (dict/scalar) | Works but untyped. Needs optional `model` parameter for typed returns. |
| `ConfigAPI.get_app_config()` / `aget_app_config()` | Returns full `AppConfig` | Potentially broken (`aget("")` path). User doesn't want full-config fetch anyway. |
| `ConfigManager._validate_config()` | Uses `AppConfig.model_validate()` only | Does not use `config_schema.json` for validation. |
| Service usage | `config_api.aget(ConfigKeys.services.tts, default={})` → raw `dict` | Correct pattern, but return is untyped `dict`, not Pydantic model. |
| `gateway/service.py`, `tts/service.py` | Use `ConfigKeys` without importing it | Would crash at runtime. |

## 4. Generation Pipeline

### 4.1 Single generation script

Everything lives in one script. No separate files for keys, defaults, etc. The script knows Aurora's paths, conventions, and requirements — nothing is parameterized.

```python
# scripts/generate_config_artifacts.py
"""Regenerate all config artifacts from config_schema.json.

This is Aurora-specific: paths, base class, codegen options are all
hardcoded business logic. Run via `make generate-config`.
"""
from __future__ import annotations

import json
import subprocess
import sys
from pathlib import Path

ROOT = Path(__file__).resolve().parent.parent
SCHEMA = ROOT / "app/services/config/config_schema.json"
MODELS_OUT = ROOT / "app/shared/config/models.py"
KEYS_OUT = ROOT / "app/shared/config/keys.py"
DEFAULTS_OUT = ROOT / "app/services/config/config_defaults.json"


def generate_models():
    """Schema → Pydantic models via datamodel-code-generator."""
    subprocess.run(
        [
            sys.executable, "-m", "datamodel_code_generator",
            "--input", str(SCHEMA),
            "--input-file-type", "jsonschema",
            "--output", str(MODELS_OUT),
            "--use-default",
            "--field-constraints",
            "--use-field-description",
            "--enum-field-as-literal", "all",
            "--base-class", "app.shared.config.models_base.BaseConfigModel",
        ],
        check=True,
    )


def generate_keys():
    """Schema → nested ConfigKeys object (every dot-path in the schema tree)."""
    with open(SCHEMA) as f:
        schema = json.load(f)

    def _walk(node, prefix=""):
        for key, sub in node.get("properties", {}).items():
            path = f"{prefix}.{key}" if prefix else key
            yield path
            if sub.get("type") == "object":
                yield from _walk(sub, path)

    paths = sorted(_walk(schema))
    lines = [
        "from enum import Enum",
        "",
        "",
        "class ConfigKeys(str, Enum):",
        '    """Auto-generated from config_schema.json. Do not edit — run `make generate-config`."""',
        "",
    ]
    for path in paths:
        name = path.upper().replace(".", "_").replace("-", "_")
        lines.append(f'    {name} = "{path}"')

    KEYS_OUT.write_text("\n".join(lines) + "\n")


def generate_defaults():
    """Generated models → config_defaults.json (instantiate with all defaults, dump)."""
    from app.shared.config.models import Model as AppConfig

    defaults = AppConfig().model_dump(mode="json")
    with open(DEFAULTS_OUT, "w") as f:
        json.dump(defaults, f, indent=2)
        f.write("\n")


def format_generated():
    """Run ruff on generated files."""
    targets = [str(MODELS_OUT), str(KEYS_OUT)]
    subprocess.run([sys.executable, "-m", "ruff", "format", *targets], check=True)
    subprocess.run([sys.executable, "-m", "ruff", "check", "--fix", *targets], check=True)


def main():
    generate_models()
    generate_keys()
    generate_defaults()
    format_generated()
    print("✓ All config artifacts regenerated from config_schema.json")


if __name__ == "__main__":
    main()
```

**Base class (`app/shared/config/models_base.py`, hand-maintained, tiny):**

```python
from pydantic import BaseModel, ConfigDict

class BaseConfigModel(BaseModel):
    model_config = ConfigDict(extra="ignore")
```

### Phase 4: Typed `ConfigAPI.aget()` with model parameter

**New signature (backward-compatible):**
```python
@overload
async def aget(self, key: str | ConfigKeys, default: Any = None, *,
               config_timeout: float = 5.0) -> Any: ...
@overload
async def aget(self, key: str | ConfigKeys, model: type[T], *,
               config_timeout: float = 5.0, default: T | None = None) -> T: ...

async def aget(self, key, model=None, *, config_timeout=5.0, default=None):
    raw = await self._fetch_section(key, config_timeout)
    if model is not None:
        if raw is None or raw == {}:
            return default if default is not None else model()
        return model.model_validate(raw)
    return raw if raw is not None else default
```

## 6. ConfigKeys generation strategy

The generated nested path object will have **every path** in the schema, at all levels:

```python
class ConfigPath(str):
    """String-compatible typed config path."""


class _ServicesTtsConfigPath(ConfigPath):
    """Auto-generated from config_schema.json. Run `make generate-config` to rebuild."""

    enabled: ConfigPath
    mesh_sharing: _ServicesTtsMeshSharingConfigPath


class _ConfigKeys:
    services: _ServicesConfigPath


ConfigKeys = _ConfigKeys()
```

Services choose the granularity they need:
- Section-level: `aget(ConfigKeys.services.tts, TtsConfig)` → full typed object
- Leaf-level: `aget(ConfigKeys.services.tooling.mcp.enabled, default=False)` → `bool`

## 7. Developer workflow

### Editing configuration schema

1. Edit `app/services/config/config_schema.json` (add field, change constraint, etc.)
2. Run `make generate-config`
3. Generated files update: `models.py`, `keys.py`, `config_defaults.json`
4. Update service code to use new keys/models if needed
5. Commit schema + generated files together

### CI protection

The `config-sync` job in `.github/workflows/lint.yml` runs on every PR:

```
make generate-config
git diff --exit-code models.py keys.py config_defaults.json
```

If a developer edits the schema but forgets `make generate-config`, CI fails with a clear error message pointing them to the command.

## 8. Migration of user `config.json` files

Users may have `config.json` files on the **old** layout (`general.*`). The migration script `scripts/migrate_config_to_services_layout.py` handles this:

1. Detect if `config.json` has `general` top-level key (old) vs `services` (new).
2. If old: remap all paths, save backup as `config.old.json`, write new layout.
3. If already new: no-op.

This script already exists (untracked). Verify it works and add `make migrate-config` target.

## 9. Risk assessment

| Risk | Mitigation |
|------|------------|
| Rewriting 1255-line schema is error-prone | Use `config_defaults.json` as structural reference; validate with `jsonschema.validate(defaults, schema)` after writing each section. |
| `datamodel-codegen` output might need tweaks (naming, `extra='ignore'`) | Use `--base-class BaseConfigModel` so all models inherit `extra='ignore'`. Review output after first generation. |
| MCP `servers` uses `patternProperties` + `allOf`/`if`/`then` which `datamodel-codegen` may not fully support | Test codegen output for MCP section. If needed, manually patch that section in a post-generation step or model MCP servers as `dict[str, Any]` with runtime validation via JSON Schema. |
| Large generated nested `ConfigKeys` object (100+ entries) | Acceptable — services import only what they need. IDE autocomplete helps navigation. |
| Circular import: generated `models.py` imported during `ConfigManager.__init__` | Already works today; no change. Keep `models_base.py` dependency-free. |
| Breaking change for services using `dict.get()` on config results | Backward-compatible: `aget(key, default={})` still returns dict. Services migrate incrementally. |
